# ADR-0008: Hardcode condition_explanation strings in strategy_engine.py

**Date:** 14Jun2026
**Status:** Accepted
**Author:** Solution Architect
**Feature:** PRD-01 — Remove Fit Scoring / Strategy Comparison Matrix

---

## Context

PRD-01 introduces a `condition_explanation` field on every `MatrixRow` object in the Strategy Comparison Matrix. This field is a single educational sentence that explains, per strategy, why the strategy's textbook-defined mechanical design criteria do or do not align with the current ticker's market conditions.

Two implementation approaches were evaluated:

**Option A — Hardcoded static strings in `strategy_engine.py`**
Each of the 31 strategies in the `STRATEGIES` catalog receives two new fields: `designed_for_iv` (`"high" | "low" | "any"`), `designed_for_direction` (`"bullish" | "bearish" | "neutral" | "volatile" | "any"`), and `condition_explanation` (a single fixed English sentence per strategy). These values are authored once at development time and never change at runtime.

**Option B — Dynamically generated by `interpreter.py` per request**
The `interpreter.py` module's `_why_this_strategy()` function (or a new sibling function) generates a sentence per strategy at analysis time, incorporating the current ticker's actual IV rank and bias values into the explanation text.

---

## Decision

Option A — Hardcoded static strings.

---

## Rationale

**Regulatory defensibility (primary driver)**

The MAS two-stage test and the SEC robo-adviser analysis in the spec both hinge on whether the output constitutes personalised investment advice. A hardcoded educational sentence is textbook fact that does not vary by ticker and does not incorporate the user's personal situation. It is materially equivalent to a printed options education textbook or a brokerage knowledge base article. Dynamic generation by `interpreter.py` introduces ticker-specific values (current IV rank, current price, current bias strength) into the text, which edges the output toward the "personalised" half of the test. Keeping the explanation text fixed removes that risk entirely.

**No Claude API dependency**
Option B would require a Claude API call per strategy row per analysis request, or at minimum require `interpreter.py` to run per-strategy. The matrix contains up to 31 rows. Even using the existing rule-based `interpreter.py` rather than the Claude API, generating 31 text blobs per request adds latency and complexity for zero gain in regulatory safety. Option A adds zero latency: the strings are already in memory as catalog constants.

**Determinism and auditability**
Option A is fully deterministic: two requests for the same ticker at different times return identical `condition_explanation` strings (holding all else equal). This is explicitly required by AC-6.7 in the spec. Option B risks text drift — minor changes to `interpreter.py` logic or upstream data would silently change the explanation strings without a schema migration, making QA verification more difficult.

**Testability**
QA can assert the exact `condition_explanation` string for each strategy key in unit tests. The 31 strings are checked into source control alongside the catalog, making regression detection trivial.

**Simplicity**
Option A adds 3 new keys to each entry in the existing `STRATEGIES` dict. No new functions, no new service calls, no async coordination. The total implementation risk is low.

---

## Rejected alternative: interpreter.py dynamic generation

Option B was rejected primarily on regulatory grounds but also on grounds of unnecessary complexity. The `_why_this_strategy()` function already uses directive language such as "ranks as the best-fit strategy" in its else-branch fallback (line 388 of `interpreter.py`). Using it as the source of `condition_explanation` would either require auditing and rewriting that function (which is out of scope for PRD-01 per the spec's OQ-3 note) or risk introducing regulatory trigger language into the matrix. The risk is asymmetric: no upside from dynamic generation, clear downside.

---

## Consequences

- The 31 `condition_explanation` strings must be authored by the developer implementing PRD-01. They must use factual, non-directive language (e.g. "Covered calls collect premium when implied volatility is elevated; this strategy is designed for HIGH IV environments").
- A future PRD that wishes to update the explanation text must make a code change to `strategy_engine.py` and deploy; it is not a database-only change.
- The `designed_for_iv` and `designed_for_direction` fields are also hardcoded in the catalog, meaning condition matching logic (`iv_condition_match`, `direction_condition_match`) can be computed in pure Python without any LLM or external service.
- `interpreter.py` is not modified by PRD-01. Its `_why_this_strategy()` section 3 narrative content (which does contain directive language) remains a PRD-05 scope item.
